# /save — 锚定当前上下文

回顾本次对话，将隐性决策和工作状态沉淀到项目文件中。

## 执行步骤

1. **创建目录**（如不存在）：
   - `mkdir -p .claude/active/`
   - `mkdir -p .claude/docs/planning/`
   - `mkdir -p .claude/docs/archive/`

2. **回顾对话**，提取以下信息：
   - 当前任务是什么（如果对话中没有明确的工作任务，跳过保存）
   - 做了哪些工作，完成了什么，还剩什么
   - 做了哪些决策（选了什么方案、放弃了什么、为什么）
   - 发现了哪些约束（不能做什么、必须做什么）
   - 有什么阻塞或待解决问题
   - 下一步具体动作

3. **覆写** `.claude/active/state.md`，严格使用以下格式（≤30 行）：

```markdown
## 当前任务
<一句话描述当前在做什么>

## 进度
- [x] 已完成项
- [ ] 未完成项

## 阻塞
<阻塞项，无则写"无">

## 下一步
<下次开始时应该做的第一件事>
```

4. **追加**新决策到 `.claude/docs/archive/decisions.md`。只记录本次对话中**新产生的**决策，不重复已有记录。每条决策使用以下格式追加到文件末尾：

```markdown
## YYYY-MM-DD <决策标题>

**Context**: <为什么需要这个决策>
**Decision**: <选了什么>
**Alternatives**: 
- <备选方案 1>: <为什么不选>
- <备选方案 2>: <为什么不选>

**Rationale**: 
- <选择的理由 1>
- <选择的理由 2>

**Consequences**: 
- <这个决策的影响>

---
```

如果本次对话没有产生新决策，不追加任何内容。

5. **更新**约束到 `.claude/docs/planning/constraints.md`（如果发现新约束）。约束包括：
   - 技术限制（"不能用 X 因为 Y"）
   - 业务规则（"必须满足 Z"）
   - 团队规范（"要遵循 W"）

6. **智能沉淀建议**（可选，默认启用）：
   
   分析对话，检测值得沉淀到 docs/ 的内容：
   
   - **决策信号**："选择 X 而非 Y"、"采用 A 放弃 B"、有明确的 tradeoff
   - **约束信号**："不能用 X 因为 Y"、"必须满足 Z"、影响未来决策
   - **理解信号**："原来 X 是因为 Y"、"X vs Y 的区别"、有深层洞察
   
   如果检测到信号，输出建议：
   
   ```
   💡 检测到可沉淀内容：
   
   【决策】<标题>
     选择：<X>
     放弃：<Y>
     理由：<why>
     → 建议追加到 docs/archive/decisions.md
   
   【约束】<类型>
     内容：<constraint>
     来源：<source>
     → 建议更新 docs/planning/constraints.md
   
   是否沉淀？[y/n]
   ```
   
   **注意**：
   - 如果用户在对话中明确说"只保存 state"或"不要建议"，跳过此步骤
   - 如果没有检测到信号，直接跳过
   - 建议数量控制在 2-5 条，避免过载
   - 只建议**提炼后的结论**，不是原始对话

7. **输出确认**，用中文简要说明保存了什么（2-3 行即可）。

## 边界情况

- **对话中没有实质性工作**（刚开始、只是闲聊）：不写任何文件，直接告知"本次对话没有需要保存的上下文"
- **没有新决策**：正常更新 active/state.md，但不追加 docs/archive/decisions.md
- **首次使用**：创建目录和文件后，提示用户将 `.claude/active/` 和 `.claude/docs/` 加入 git 跟踪

## 注意事项

- `active/state.md` 每次**覆写**，永远反映最新状态
- `docs/archive/decisions.md` 只**追加**，不修改已有内容
- `docs/planning/constraints.md` 可以**更新**（新增或修改约束）
- 提取决策时关注"为什么"而非"做了什么"——做了什么在 git 里有，为什么不记就丢了
- 智能建议是辅助功能，不是强制的——如果建议不准确，用户可以拒绝
- **不要**在确认输出中建议用户运行其他命令（如 /recap），避免触发自动执行

## 判断标准（智能建议）

**值得沉淀**：
- 决策有明确的 tradeoff（不是显而易见的选择）
- 约束会影响未来决策（不是一次性限制）
- 理解有深度（不是表面操作说明）
- 讨论深度 >3 轮对话

**不值得沉淀**：
- 纯操作记录（"创建了 X 文件"）
- 临时状态（"测试通过了"）
- 显而易见的选择（"用 Git 做版本控制"）
- 可从代码推断的信息（"用了 React"）
