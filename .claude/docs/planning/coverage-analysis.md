# 深度反思：devkit 的覆盖度与实战差距

## 当前设计的假设

devkit 基于以下假设构建：
1. **单人开发者**：个人项目，多任务切换
2. **会话粒度**：以"一次对话"为单位沉淀
3. **线性演进**：决策 → 实现 → 下一个决策
4. **轻量优先**：极简主义，2 个命令

## 真实开发场景的复杂性

### 场景 1：探索式开发（Spike）

**现实**：
```
尝试方案 A（2 小时）→ 不行，换 B
尝试方案 B（3 小时）→ 也不行，回到 A 的变体
尝试方案 A'（1 小时）→ 可行！
```

**devkit 的问题**：
- ❌ 只记录最终决策（"选了 A'"），丢失了探索过程
- ❌ 不知道 B 为什么不行（具体卡在哪）
- ❌ 如果 3 个月后遇到类似问题，可能重复踩坑

**其他 builder 的实践**：
- **Spike 日志**：记录每次尝试的假设、实验、结果
- **失败档案**：专门记录"什么不行"（比"什么行"更有价值）
- **时间线追踪**：按时间顺序记录思考演进

### 场景 2：多分支并行开发

**现实**：
```
main: 稳定版本
feature/auth: 实现认证（进行中）
feature/api: 重构 API（暂停，等 auth）
hotfix/bug-123: 紧急修复（插队）
```

**devkit 的问题**：
- ❌ `active/state.md` 只有一个，无法表达多分支状态
- ❌ 切换分支时，state 会被覆盖
- ❌ 不知道 feature/api 为什么暂停，暂停前做到哪了

**其他 builder 的实践**：
- **分支级 context**：每个分支有自己的 state
- **依赖追踪**：记录分支间的依赖关系
- **暂停原因**：记录为什么暂停，什么条件下恢复

### 场景 3：渐进式重构

**现实**：
```
Week 1: 发现代码腐化，决定重构
Week 2: 重构 20%，发现影响面比预期大
Week 3: 调整策略，改为分阶段重构
Week 4: 完成第一阶段
```

**devkit 的问题**：
- ❌ 决策在演进（从"一次性重构"到"分阶段"），但 decisions.md 只追加，不表达演进
- ❌ 不知道当前在哪个阶段，下一阶段是什么
- ❌ 重构的"为什么"分散在多次对话中，难以整体回顾

**其他 builder 的实践**：
- **Epic/Story 层级**：大任务拆分为多个小任务，追踪整体进度
- **决策演进图**：可视化决策如何随时间变化
- **阶段里程碑**：明确当前在哪个阶段，完成标准是什么

### 场景 4：技术债务管理

**现实**：
```
发现问题：API 设计不一致
评估影响：影响 15 个端点
决策：暂不修复（优先级低）
3 个月后：技术债累积，必须还了
```

**devkit 的问题**：
- ❌ 没有"已知问题清单"（不是阻塞，但需要记住）
- ❌ 不知道技术债的优先级和影响面
- ❌ 3 个月后可能忘记当初为什么不修复

**其他 builder 的实践**：
- **Tech Debt Register**：专门的技术债清单
- **优先级矩阵**：影响面 × 紧急度
- **定期回顾**：每月/季度回顾技术债

### 场景 5：团队协作

**现实**：
```
我：实现了 feature X，决策是 A
同事：review 后建议改为 B
我：接受，但需要调整其他模块
同事：我来调整，你继续下一个
```

**devkit 的问题**：
- ❌ 决策是"我"做的还是"团队"做的？
- ❌ 不知道谁负责什么
- ❌ 同事看不到我的 `.claude/`（在本地）

**其他 builder 的实践**：
- **Decision Owner**：每个决策标注负责人
- **Shared Context**：团队共享的决策库（Notion/Confluence）
- **Review Trail**：记录 review 中的讨论和修改

## 网络上 builder 的实践模式

### 模式 1：可溯源的文档（Traceability）

**代表**：Linear、Notion 的 "Connected Docs"

**特点**：
- 每个决策/任务有唯一 ID
- 文档间可以互相引用（`DEC-001` 引用 `TASK-042`）
- 可以追溯"为什么做这个任务"（链到决策）

**devkit 缺失**：
- ❌ 没有 ID 系统
- ❌ 决策和任务是分离的（decisions.md vs state.md）
- ❌ 无法表达"这个任务是因为那个决策"

### 模式 2：分层文档（Hierarchical）

**代表**：Obsidian 的 "MOC (Map of Content)"

**特点**：
```
README.md (入口)
  ├─ Architecture/ (架构层)
  │   ├─ Overview.md
  │   └─ Decisions/
  │       ├─ 001-use-postgres.md
  │       └─ 002-api-design.md
  ├─ Features/ (功能层)
  │   ├─ Auth/
  │   └─ API/
  └─ Operations/ (运维层)
      ├─ Deployment.md
      └─ Monitoring.md
```

**devkit 缺失**：
- ❌ 只有 2 层（active vs docs），没有更细的分类
- ❌ 所有决策在一个文件（decisions.md），难以按主题浏览
- ❌ 没有"入口文档"（README for context）

### 模式 3：时间线 + 快照（Timeline + Snapshots）

**代表**：Roam Research 的 "Daily Notes"

**特点**：
- 每天一个笔记（2026-04-25.md）
- 记录当天的思考、决策、进度
- 可以回溯"3 个月前在想什么"

**devkit 缺失**：
- ❌ state.md 覆写，丢失历史快照
- ❌ 依赖 git history，但 git 是代码视角，不是思考视角
- ❌ 无法快速回答"上周做了什么决策"

### 模式 4：标签 + 搜索（Tags + Search）

**代表**：Notion 的 Database + Tags

**特点**：
- 每个决策/任务有标签（#auth #api #refactor）
- 可以按标签过滤（"所有关于 auth 的决策"）
- 可以全文搜索

**devkit 缺失**：
- ❌ 没有标签系统
- ❌ 搜索依赖 grep（不智能）
- ❌ 无法回答"所有关于性能的决策"

### 模式 5：模板 + 自动化（Templates + Automation）

**代表**：GitHub Issues 的 Templates

**特点**：
- 决策有模板（Context/Decision/Alternatives）
- 自动生成 ID、时间戳
- 自动关联相关 issue/PR

**devkit 缺失**：
- ✅ 有模板（ADR 格式）
- ❌ 没有自动化（手动写入）
- ❌ 没有关联（决策和代码分离）

## 深度反思：devkit 的定位问题

### 当前定位：个人开发者的"会话记忆"

**适合场景**：
- ✅ 单人项目
- ✅ 简单决策（A vs B）
- ✅ 短期记忆（几天到几周）

**不适合场景**：
- ❌ 团队协作
- ❌ 复杂探索（多次尝试）
- ❌ 长期演进（几个月到几年）
- ❌ 大型项目（多模块、多分支）

### 核心问题：粒度不匹配

**devkit 的粒度**：会话（1-2 小时对话）

**真实开发的粒度**：
- **Spike**：几小时到几天的探索
- **Feature**：几天到几周的开发
- **Epic**：几周到几个月的大任务
- **Project**：几个月到几年的项目

**结果**：devkit 只覆盖了最细粒度，缺少中间层和顶层。

## 可能的演进方向

### 方向 1：保持极简，明确边界

**策略**：
- 承认 devkit 只解决"会话级记忆"
- 不试图解决所有问题
- 与其他工具配合（Linear/Notion 管理 Epic，devkit 管理会话）

**优点**：
- 保持简单
- 专注核心价值

**缺点**：
- 覆盖度有限
- 需要用户自己整合多个工具

### 方向 2：渐进增强，分层架构

**策略**：
```
.claude/
├─ active/           # 会话级（现有）
│   └─ state.md
├─ docs/             # 项目级（现有）
│   ├─ planning/
│   └─ archive/
├─ features/         # 功能级（新增）
│   ├─ auth/
│   │   ├─ state.md
│   │   └─ decisions.md
│   └─ api/
└─ project/          # 项目级（新增）
    ├─ roadmap.md
    ├─ tech-debt.md
    └─ timeline.md
```

**优点**：
- 覆盖更多场景
- 保持向后兼容

**缺点**：
- 复杂度增加
- 违背极简主义

### 方向 3：插件化，核心 + 扩展

**策略**：
- 核心：/save + /recap（现有）
- 扩展：/spike（探索日志）、/debt（技术债）、/timeline（时间线）
- 用户按需启用

**优点**：
- 灵活
- 不强制复杂度

**缺点**：
- 维护成本高
- 用户需要学习多个命令

## 我的判断

### 当前 devkit 的价值

**已覆盖**（80% 的个人开发场景）：
- ✅ 多任务切换（项目 A/B/C）
- ✅ 中断恢复（几天后回来）
- ✅ 简单决策记录（A vs B）

**未覆盖**（20% 的复杂场景）：
- ❌ 探索式开发（多次尝试）
- ❌ 多分支并行
- ❌ 长期演进追踪
- ❌ 团队协作

### 建议的演进路径

**Phase 1（当前）**：
- 保持极简，验证核心价值
- 在实际项目中 dogfood
- 收集真实痛点

**Phase 2（3 个月后）**：
- 根据 dogfood 结果，决定是否扩展
- 如果核心价值验证成功，考虑添加：
  - Spike 日志（探索过程）
  - 分支级 state（多分支）
  - 决策演进追踪（决策如何变化）

**Phase 3（6 个月后）**：
- 如果用户增长，考虑团队协作功能
- 如果保持小众，专注个人开发者

### 关键洞察

**不要试图成为"万能工具"**：
- Notion：适合结构化知识管理
- Linear：适合任务和项目管理
- Obsidian：适合个人知识库
- **devkit**：适合 AI 对话的上下文沉淀

**devkit 的独特价值**：
- 与 Claude Code 深度集成
- 自动从对话中提取（不是手动记录）
- Git 友好（纯文本，可 diff）

**不要复制其他工具**：
- 如果用户需要复杂的项目管理 → 用 Linear
- 如果用户需要知识图谱 → 用 Obsidian
- devkit 专注于"对话 → 资产"的转换

## 结论

**当前设计能覆盖 80% 的个人开发场景**，但有明显的边界：

**适合**：
- 单人项目
- 简单到中等复杂度
- 短期到中期（几天到几周）

**不适合**：
- 团队协作
- 复杂探索
- 长期演进（几个月到几年）

**建议**：
1. 先验证核心价值（当前设计）
2. 在实战中发现真实痛点
3. 根据痛点决定是否扩展
4. 保持"对话 → 资产"的核心定位，不要试图成为万能工具
