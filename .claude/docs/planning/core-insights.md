# 核心洞察：专注的活动空间 + 累积的文件事实

## 认知模型演进

### 初始问题
AI 对话产生大量隐性知识（决策、约束、理解），但会话结束后流失。现有机制：
- `--continue`：恢复会话，但不适合跨项目
- `memory`：记住偏好，但不捕获决策 why
- Git：记录代码变更，但不记录思考过程

### 核心洞察

**"专注的活动空间 + 累积的文件事实"**

这是两种不同性质的信息存储：

| 维度 | 活动空间（Active） | 文件事实（Docs） |
|------|-------------------|-----------------|
| **本质** | 工作记忆 | 长期记忆 |
| **类比** | 大脑 + 草稿纸 | 笔记本 + 档案袋 |
| **特征** | 易变、会话级、可丢弃 | 稳定、项目级、持久化 |
| **操作** | 覆写（反映最新状态） | 追加/更新（累积知识） |
| **内容** | 当前任务、进度、阻塞 | 决策、约束、架构理解 |
| **生命周期** | 随会话结束而过期 | 跨会话持续有效 |

### 进一步分层：Planning vs Archive

在"文件事实"内部，还有时态的区分：

| 维度 | Planning（笔记本） | Archive（档案袋） |
|------|-------------------|------------------|
| **时态** | 现在/未来 | 过去 |
| **可变性** | 可更新、可覆写 | 只追加、不修改 |
| **内容** | 路线图、约束、架构 | 决策日志、探索记录 |
| **用途** | 指导未来决策 | 解释历史选择 |

## 实现映射

```
认知模型                    文件结构
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
活动空间                    .claude/active/
  └─ 当前状态                 └─ state.md

文件事实                    .claude/docs/
  ├─ 笔记本（活跃）            ├─ planning/
  │   ├─ 路线图                │   ├─ roadmap.md
  │   ├─ 约束                  │   ├─ constraints.md
  │   └─ 架构理解              │   └─ architecture.md
  └─ 档案袋（历史）            └─ archive/
      └─ 决策日志                  └─ decisions.md
```

## 关键设计决策

### 1. 为什么不用 memory/?

**决策**：使用 `docs/` 而非 `memory/`

**理由**：
- `docs/` 是开发者熟悉的约定（降低心智负担）
- 暗示"文档化的知识"而非"原始记忆"
- 与 `active/`（工作区）形成清晰对比

### 2. 为什么分 Planning 和 Archive?

**决策**：分为 `planning/`（活跃文档）和 `archive/`（历史记录）

**理由**：
- 对应"笔记本/档案袋"的认知模型
- 时态清晰：现在/未来 vs 过去
- 操作语义：更新 vs 追加
- 避免"所有东西都追加"导致的文件膨胀

### 3. 为什么 state.md 要覆写？

**决策**：`active/state.md` 每次覆写，不保留历史

**理由**：
- 活动空间的本质是"当前状态"，不是"状态历史"
- 历史在 git commit 里，不需要重复存储
- 保持文件简洁（≤30 行），快速扫描

### 4. 为什么 decisions.md 只追加？

**决策**：`docs/archive/decisions.md` 只追加，不修改已有记录

**理由**：
- 决策是历史事实，修改会丢失演进轨迹
- ADR（Architecture Decision Records）的标准实践
- 如果决策被推翻，应该追加新决策说明为什么改变

## 信息流动

```
对话中的隐性知识
    ↓
【检测信号】
  - 决策信号："选择 X 而非 Y"
  - 约束信号："不能用 X 因为 Y"
  - 理解信号："原来 X 是因为 Y"
    ↓
【智能建议】
  - 提炼结论（不是原始对话）
  - 过滤显而易见的内容
  - 控制数量（2-5 条）
    ↓
【用户确认】
  - 接受：写入对应文件
  - 拒绝：跳过
    ↓
【持久化】
  - active/state.md      ← 覆写
  - docs/archive/        ← 追加
  - docs/planning/       ← 更新
    ↓
【Git 跟踪】
  - 跨设备同步
  - 版本历史
    ↓
【跨会话恢复】
  - /recap 读取 active + docs + git
  - 输出简报
  - 快速回到工作状态
```

## 与现有机制的关系

| 机制 | 作用域 | 持久性 | 用途 | 互补关系 |
|------|--------|--------|------|----------|
| `--continue` | 单会话 | 临时 | 恢复中断的对话 | devkit 处理跨会话 |
| `memory` | 跨会话 | 永久 | 记住用户偏好 | devkit 记住项目决策 |
| Git | 项目级 | 永久 | 代码版本控制 | devkit 记录思考过程 |
| **devkit** | 项目级 | 永久 | 决策和上下文 | 填补"为什么"的空白 |

## 设计原则

1. **极简主义**：核心命令不超过 3 个（实际 2 个）
2. **Git 友好**：纯文本 Markdown，可 diff，可同步
3. **人类可读**：不依赖工具也能理解
4. **渐进增强**：基础功能（保存/恢复）+ 可选增强（智能建议）
5. **失败优雅**：建议不准确？用户可以拒绝；忘记 /save？有自动提醒

## 核心价值主张

**从对话到资产**：
- 对话是易逝的（会话结束即消失）
- 资产是持久的（git 跟踪，跨会话复利）
- devkit 是转换器（将对话中的隐性知识转化为显性资产）

**三个核心场景**：
1. **多任务切换**：在项目 A/B/C 间切换时，/recap 快速恢复
2. **中断恢复**：几天后回到项目，记得"为什么这样做"
3. **知识复利**：决策和约束沉淀后，避免重复踩坑
